# Web UI

This directory contains the lightweight web UI client, rendering, and styling helpers.

- Keep UI concerns here and avoid leaking server or workflow logic into presentation code.
- Favor clear rendering behavior over framework-heavy abstractions.
- `client.ts` and `styles.ts` are assembly files that compose section modules (`client-*.ts`, `styles-*.ts`) into the final `WEB_UI_JS` / `WEB_UI_CSS` exports. Add new UI behaviour in a focused section module rather than growing the assemblers. Key sections: `client-sessions.ts` (chat session list and conversation history — not the daemon sessions panel), `client-active-sessions.ts` (live daemon sessions panel showing active `kota serve` sessions from `GET /api/daemon/status`), `client-chat.ts` (message rendering + send), `client-workflows.ts` (workflow controls, history filter, and browser push notification setup via `initBrowserNotifications`), `client-run-detail.ts` (core run detail: `showRunDetail` fetches run data, artifacts, thinking, and triggered runs then calls `renderRunDetail`; `renderRunDetail` assembles the header/meta/warnings/artifacts HTML, delegates step rows to `renderStepRowsHtml` and event binding to `bindRunDetailControls`; `_openRunFromHash` restores the panel from `#run=<runId>` on page load — called from `client.ts` after `refreshWorkflows()`), `client-run-detail-controls.ts` (abort/retry/replay/resume-from-step buttons: `scrollToApprovals`, `closeStream`, `showChat`, `resumeFromStep`, and `bindRunDetailControls` which wires all button click handlers and show-more/less toggles after the run detail HTML is set), `client-run-detail-steps.ts` (step list rendering: `renderStepRowsHtml` returns the step-progress indicator, log-search bar, and all step rows HTML including foreach/branch/approval-waiting compound blocks; `toggleThinking`, `toggleForeachDetail`, `applyLogSearch`), `client-run-detail-stream.ts` (SSE streaming: `startRunStream` opens the run event stream and `handleStreamEvent` processes `step_started`/`step_output`/`step_tool`/`step_thinking`/`step_completed`/`run_completed` events), `client-run-detail-compare.ts` (run diff panel: `_compareRunId` state, `renderCompareSection` builds the compare dropdown, `renderDiffTable` renders a side-by-side step comparison table with duration and cost deltas), `client-tasks.ts` (normalized task queue panel grouped by state — doing/ready/backlog/blocked — with per-task action buttons to move between states or drop, plus an inline capture form that writes a lightweight inbox note via `POST /api/tasks`; queue data still comes from `GET /api/tasks` and body/state edits use `PATCH /api/tasks/:id/*`), `client-approvals.ts`, `client-cost.ts` (cost analytics panel — time-window selector, per-workflow totals, top-runs list, and a compact inline SVG bar chart showing daily or hourly spend: 24 bars for the 24h window, day-by-day bars for 7d/30d; chart is driven by the same run fetch as the totals, no extra API calls), `client-schedules.ts` (schedules panel showing workflows with cron/interval triggers, next-run time, and last-run status from `GET /api/workflow/status` + `GET /api/workflow/definitions`), `client-extensions.ts` (extensions panel showing loaded extensions from `GET /api/extensions` with per-extension health badge color-coded by status: ok/restarting/dead), `client-knowledge.ts` (knowledge store browser panel with text filter, inline add form, and per-row delete button — `GET/POST/DELETE /api/knowledge`), `client-memory.ts` (memory store browser panel with text filter, inline add form, and per-row delete button — `GET/POST/DELETE /api/memory`), `client-theme.ts` (light/dark theme toggle — reads/writes `kota.theme` in localStorage, toggles `light` class on `<body>`), `client-keyboard.ts` (keyboard navigation — j/k to cycle run history, Escape to close run detail, / to focus log search), `client-status-overview.ts` (system status overview panel rendered at the top of the sidebar — daemon uptime from `GET /api/daemon/status`, dispatch window state, 24h cost from `_allRecentRuns`, last-1h run health counts, extension health summary from `_cachedExtensions` set by `client-extensions.ts`; `refreshOverview()` is called after `refreshWorkflows` resolves and on SSE queue events; `_cachedExtensions` is a shared IIFE-level variable populated by `client-extensions.ts` to avoid a duplicate fetch), `client-utils.ts`, `client-audit.ts` (guardrail audit trail panel with risk and policy filters from `GET /api/audit`), `client-config.ts` (config viewer panel — fetches `GET /api/config` on load and renders each top-level key/value row; sensitive values already masked server-side), `client-wf-definitions.ts` (all-workflows definitions panel showing trigger type, last status, last run time, a compact "Inputs: field*: type, ..." line when a workflow declares `inputSchema`, and a compact "Outputs: field: type, ..." line when a workflow declares `outputSchema`; clicking Run on a workflow with a non-empty `inputSchema` shows an inline form collecting required and optional fields before triggering — required fields are validated before `triggerWorkflowByName` is called with the collected payload; each row has an Enable/Disable toggle button that calls `POST /api/workflow/definitions/:name/enable` or `/disable` and refreshes the panel — the effective enabled state accounts for `runtimeEnabled` override when present, from `GET /api/workflow/definitions` + `GET /api/workflow/status`). `styles-theme.ts` contributes the `body.light` CSS override block that re-declares all color custom properties for the light theme.

## Responsive Layout

The layout adapts at two breakpoints defined in `styles-layout.ts`:
- **≤768px (tablet)**: the sidebar becomes a fixed overlay that slides in/out. `#mobile-menu-btn` (hidden on desktop) and `#sidebar-overlay` (backdrop) are rendered in `web-ui.ts`. `client.ts` wires their click handlers and auto-collapses the sidebar on load when `window.innerWidth <= 768`.
- **≤480px (mobile)**: single-column layout, no horizontal overflow.

Desktop layout (≥900px) is unchanged. When adding new layout changes, keep media queries in `styles-layout.ts`; keep sidebar toggle JS in `client.ts` rather than individual section modules.

## SSE Reconnect Pattern

`connectDaemonEvents()` in `client-workflows.ts` tracks `_lastEventTimestamp` and appends `since=<timestamp>` to the `EventSource` URL on reconnect so the daemon ring buffer replays missed events immediately. Every new SSE event listener added inside `connectDaemonEvents` **must** be wrapped with `_trackEvent(handler)` — this updates `_lastEventTimestamp` on each event and ensures the reconnect window covers all event types. Omitting the wrapper means events of that type will not advance the timestamp, and catchup after reconnect may replay duplicates or miss events depending on event ordering.

## Filter Pattern

When adding client-side filters to a panel: (1) keep filter state in plain JS variables, (2) add an `apply<Panel>Filter()` function that reads state and calls the renderer with filtered data, (3) add a `render<Panel>Filter(names)` that rebuilds the filter bar and restores values from state, (4) store the full fetched dataset in a module-level array so filters work without re-fetching. The `#workflow-history-filter` element in the Workflows section is the reference implementation.
