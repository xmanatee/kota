# Web UI

This directory contains the lightweight web UI client, rendering, and styling helpers.

- Keep UI concerns here and avoid leaking server or workflow logic into presentation code.
- Favor clear rendering behavior over framework-heavy abstractions.
- `client.ts` and `styles.ts` are assembly files that compose section modules (`client-*.ts`, `styles-*.ts`) into the final `WEB_UI_JS` / `WEB_UI_CSS` exports. Add new UI behaviour in a focused section module rather than growing the assemblers. Key sections: `client-sessions.ts` (chat session list and conversation history — not the daemon sessions panel), `client-active-sessions.ts` (live daemon sessions panel showing active `kota serve` sessions from `GET /api/daemon/status`), `client-chat.ts` (message rendering + send), `client-workflows.ts` (workflow controls, history filter, and browser push notification setup via `initBrowserNotifications`), `client-run-detail.ts` (run detail + step progress), `client-tasks.ts` (task queue panel grouped by state — doing/ready/backlog/blocked — with per-task action buttons to move between states or drop, and an inline new-task form — `GET /api/tasks`, `PATCH /api/tasks/:id/state`, `POST /api/tasks`), `client-approvals.ts`, `client-cost.ts`, `client-schedules.ts` (schedules panel showing workflows with cron/interval triggers, next-run time, and last-run status from `GET /api/workflow/status` + `GET /api/workflow/definitions`), `client-extensions.ts` (extensions panel showing loaded extensions from `GET /api/extensions` with per-extension health badge color-coded by status: ok/restarting/dead), `client-knowledge.ts` (knowledge store browser panel with text filter, inline add form, and per-row delete button — `GET/POST/DELETE /api/knowledge`), `client-memory.ts` (memory store browser panel with text filter, inline add form, and per-row delete button — `GET/POST/DELETE /api/memory`), `client-theme.ts` (light/dark theme toggle — reads/writes `kota.theme` in localStorage, toggles `light` class on `<body>`), `client-keyboard.ts` (keyboard navigation — j/k to cycle run history, Escape to close run detail, / to focus log search), `client-utils.ts`, `client-audit.ts` (guardrail audit trail panel with risk and policy filters from `GET /api/audit`), `client-wf-definitions.ts` (all-workflows definitions panel showing trigger type, last status, and last run time from `GET /api/workflow/definitions` + `GET /api/workflow/status`). `styles-theme.ts` contributes the `body.light` CSS override block that re-declares all color custom properties for the light theme.

## Filter Pattern

When adding client-side filters to a panel: (1) keep filter state in plain JS variables, (2) add an `apply<Panel>Filter()` function that reads state and calls the renderer with filtered data, (3) add a `render<Panel>Filter(names)` that rebuilds the filter bar and restores values from state, (4) store the full fetched dataset in a module-level array so filters work without re-fetching. The `#workflow-history-filter` element in the Workflows section is the reference implementation.
